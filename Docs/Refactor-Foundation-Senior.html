Refactor para uma boa foundation

Análise do código atual de As Profundezas e reescrita segundo práticas senior — mantendo tudo o que discutimos exceto o Event Bus global. Cada ponto liga a um ficheiro real do teu projeto, com "antes" e "depois".

A regra desta sessão: sem Event Bus — mas atenção à distinção

Vetaste o Event Bus global (aquele EventBus<T> estático, pub/sub para o jogo inteiro). Concordo — para o tamanho deste projeto é complexidade a mais e foi o único ponto que eu próprio marquei como "faz só quando doer".

Mas cuidado com um mal-entendido: quando eu falar em "eventos locais" na Health (secção 3), isso não é o Event Bus. É o mesmo que um onClick de um componente React: um sinal local, deste objeto, com subscritores explícitos e visíveis. Sem estado global, sem strings, sem teia invisível. Se mesmo assim preferires zero eventos, dou-te a alternativa por chamada direta.

Resumo executivo da análise

O teu código está bom para um protótipo — o padrão data-driven com ScriptableObject (Ability, AIState) é exatamente a escolha certa, e o CharacterManager como raiz de composição é sólido. Os problemas são os típicos de código que cresceu depressa: responsabilidades misturadas (a Health faz de tudo), acoplamento (quem calcula dano espreita para dentro de quem ataca), duplicação em 3 sítios (os tipos de dano) e alocações por frame (Instantiate de SOs). Nada disto é grave agora; tudo isto será grave daqui a 3 meses. É a altura certa para consolidar.

1. Tipos de dano: 5 campos hardcoded → DamageType (ScriptableObject) fazer já

Hoje os cinco tipos de dano estão hardcoded em três ficheiros diferentes. Adicionar um tipo novo (ex.: "veneno", perfeito para a alquimista Ivy) obriga a editar código C#. Isto é o oposto do "adicionar conteúdo = criar um asset".

Decisão de design (fechada)

Não usamos um enum para o tipo de dano. Usamos um ScriptableObject — o mesmo padrão que já usas em Ability e AIState. Um tipo novo = Create → asset no Project, sem recompilar. Mantém o modelo simples: um tipo + um valor de dano por golpe (sem arrays, sem pacotes multi-tipo).

Assets/Scripts/Abilities/MeleeAttack.cs — os campos 
antespublic int slashDamage = 0;
public int bluntDamage = 0;
public int pierceDamage = 0;
public int fireDamage = 0;
public int frostDamage = 0;
Assets/Scripts/DamageReceiver.cs — o cálculo (repara na inconsistência!) 
antes[Range(0f, 1f)] public float slashReduction = 1f;   // é 0..1 (percentagem)
// ...
if (attackAbility.slashDamage > 0)
    damage += Mathf.Clamp(attackAbility.slashDamage - slashReduction, 0, Mathf.Infinity);
//                                                  ^^^ subtrai um valor 0..1 a um dano inteiro
Bug real que encontrei

A slashReduction tem [Range(0,1)] — parece uma percentagem. Mas depois é subtraída como valor absoluto: slashDamage - slashReduction. Com dano 30 e redução 0.8, o resultado é 29.2 → arredonda para 29. Ou seja: as reduções praticamente não fazem nada. A intenção era certamente dano × (1 − redução). Este tipo de bug esconde-se precisamente porque a lógica está espalhada e repetida.

O "depois": DamageType como ScriptableObject

Cada tipo de dano é um asset (Slash, Blunt, Fire, Poison…). A arma/ataque aponta para um e declara quanto dano causa. O receptor procura a redução associada a esse asset.

conceito — redução no receptor (lista em vez de 5 campos) 
depois[System.Serializable]
public struct DamageResistance
{
    public DamageType type;          // referência ao mesmo SO
    [Range(0f, 1f)] public float reduction; // 0 = sem redução, 1 = imune
}

float GetReduction(DamageType type)
{
    for (int i = 0; i < resistances.Count; i++)
        if (resistances[i].type == type)
            return resistances[i].reduction;
    return 0f;

Porquê SO e não enum?
  • Conteúdo sem código — criar "Poison" é Create Asset, não editar C# e recompilar.
  • Dados ricos — nome, ícone, cor de UI vivem no asset (um enum só dá um identificador).
  • Coerente com o projecto — já pensas em dados com Ability, AIState; o tipo de dano é mais um dado.
  • Simples — um SO + um int damage. Sem arrays, sem DamagePacket multi-tipo.
Como criar os assets no Unity

Botão direito → Create → AF → Data → Damage Type. Crias um por tipo (DT_Slash, DT_Fire, DT_Poison…). Nas armas e nas resistências, arrastas a referência. Adicionar tipo novo = novo asset + arrastar onde fizer sentido.

2. O fluxo de dano: quebrar o acoplamento fazer já

Hoje quem recebe o dano espreita para dentro de quem ataca para descobrir os números:

Assets/Scripts/DamageReceiver.cs 
antespublic void ApplyDamage(CharacterManager attacker)
{
    // O receptor tem de saber que o atacante tem um combatManager,
    // que tem uma "current ability", que pode ser um MeleeAttack...
    MeleeAttack attackAbility = attacker.combatManager.GetCurrentAbility() as MeleeAttack;
    if (attackAbility != null) { /* lê slashDamage, etc. */ }
}
O smell: "feature envy" + Law of Demeter

attacker.combatManager.GetCurrentAbility() as MeleeAttack é uma cadeia de 3 saltos e um cast frágil. Se o ataque não for um MeleeAttack (uma bola de fogo? uma armadilha do Duval?), o dano é silenciosamente zero. O receptor sabe coisas a mais sobre o atacante.

O "depois": o atacante declara o que faz; o receptor resolve

Inverte-se o fluxo: em vez de o receptor ir buscar, o atacante entrega uma requisição de dano. Isto é um struct imutável — o "contrato" do golpe. Nada de bus, é uma chamada de método direta.

depoispublic readonly struct DamageRequest
{
    public readonly CharacterManager Source;
    public readonly DamageType Type;   // referência ao ScriptableObject
    public readonly int Amount;
    public readonly Vector3 HitPoint;

    public DamageRequest(CharacterManager source, DamageType type, int amount, Vector3 hitPoint)
    { Source = source; Type = type; Amount = amount; HitPoint = hitPoint; }
}

public interface IDamageable
{
    DamageResult Receive(in DamageRequest request);
}

A Hitbox, ao acertar, constrói a DamageRequest a partir da arma/ability atual (damageType + damage) e chama target.Receive(request). O receptor aplica a redução desse tipo e devolve um DamageResult. Agora:

  • Qualquer fonte de dano (melee, fogo, armadilha, veneno) usa o mesmo caminho — só mudam o asset DamageType e o valor.
  • Não há casts frágeis nem saltos de 3 níveis.
  • É testável: constróis uma DamageRequest num teste e verificas o DamageResult, sem cena, sem animação.
Precisas de uma "chain of responsibility"?

Na versão senior falei de um pipeline de estágios. Ainda não precisas. Começa com um método Receive claro que faz: 1) checar invulnerabilidade (o teu IsInvulnerable() do dodge já existe), 2) aplicar reduções, 3) devolver resultado. Quando tiveres 5+ regras (críticos, lifesteal, stagger), aí sim partes o método numa cadeia. Uma abstração de cada vez.

3. Health: estado puro + reações desacopladas (sem bus) fazer já

Este é o ponto que já tínhamos identificado. O teu Die() faz quatro trabalhos diferentes:

Assets/Scripts/Health.cs 
antesvoid Die()
{
    characterManager.animator.Play("Death");          // animação
    characterManager.agent.ResetPath();               // navegação
    characterManager.agent.enabled = false;
    characterManager.isBusy = true;                   // estado de combate
    var player = characterManager.GetPlayer();
    if (player != null) player.lockOn.UnregisterChasingEnemy(characterManager); // lock-on do player
}

Sempre que quiseres uma reação nova à morte (dropar loot/minério, dar XP ao herói — a ironia do GDD, screen shake, som), este método incha. E não consegues testar "a vida chega a 0?" sem um Animator e um NavMeshAgent na cena.

Opção A (recomendada): eventos locais — não é o Event Bus

depoispublic sealed class Health : MonoBehaviour
{
    public int Current { get; private set; }
    public int Max = 100;
    public bool IsDead => Current <= 0;

    // Sinais LOCAIS deste componente. Subscritores explícitos, sem estado global.
    public event System.Action<DamageResult> OnDamaged;
    public event System.Action OnDied;

    public void Apply(in DamageResult result)
    {
        if (IsDead) return;
        Current = Mathf.Max(0, Current - result.FinalDamage);
        OnDamaged?.Invoke(result);
        if (Current == 0) OnDied?.Invoke();
    }
}

E cada reação vira um pequeno componente no mesmo personagem, que subscreve no OnEnable e dessubscreve no OnDisable (a regra de ouro para não haver leaks):

depoispublic class DeathReactor : MonoBehaviour   // um por responsabilidade
{
    [SerializeField] CharacterManager cm;
    [SerializeField] Health health;

    void OnEnable()  => health.OnDied += HandleDeath;
    void OnDisable() => health.OnDied -= HandleDeath;   // simétrico, sempre!

    void HandleDeath()
    {
        cm.animator.Play(AnimHashes.Death);   // ver secção 6
        cm.agent.enabled = false;
        cm.isBusy = true;
    }
}
// Outros: LootDropReactor, LockOnReactor... cada um subscreve o que lhe interessa.
Porque isto não é o Event Bus que vetaste

O Event Bus global é estático: qualquer sistema do jogo emite/escuta sem referência direta, e ninguém sabe quem escuta. Aqui, os eventos pertencem a uma instância de Health, os subscritores estão no mesmo GameObject, e vês as ligações no Inspector. É observação local e explícita — a diferença é a mesma entre um onClick de um botão e um sistema de pub/sub global.

Opção B (se quiseres mesmo zero eventos)

A Health expõe só o estado (Current, IsDead, Apply) e um único componente CharacterDeathHandler — referenciado diretamente — trata da reação, chamado pela Health via uma referência serializada. Menos flexível (uma só reação, ou uma que chama as outras por ordem), mas 100% sem eventos. Recomendo a A; menciono a B porque respeita a tua preferência se a levares ao extremo.

4. Alocações por frame: parar de clonar ScriptableObjects fazer já

Isto liga diretamente à conversa que tivemos sobre NonAlloc e lixo para o Garbage Collector. Encontrei dois sítios que clonam ScriptableObjects com frequência:

Assets/Scripts/CombatManager.cs 
antespublic void SetCurrentAbility(Ability next)
{
    if (currentAbility != null) currentAbility.OnEnd(characterManager);
    currentAbility = Instantiate(next);   // ← novo objeto CADA ataque
    currentAbility?.OnStart(characterManager);
}
Assets/Scripts/StateMachine.cs 
antespublic void SwitchState(AIState newState)
{
    currentState?.Exit(this);
    currentState = Instantiate(newState);  // ← novo objeto CADA transição
    currentState?.Enter(this);
}
O problema

Um auto-atacante (Vampire Survivors!) dispara ataques constantemente. Uma IA muda de estado várias vezes por segundo × dezenas de inimigos. Cada Instantiate de um SO cria um objeto novo que o GC terá de limpar → stutters periódicos em mobile. Provavelmente clonaste para teres estado por-instância (ex.: lastAttackTime) sem sujar o asset partilhado — instinto correto, solução cara.

O "depois": separar dados imutáveis de estado de runtime

O ScriptableObject é dados partilhados e só-de-leitura (o "flyweight"). O estado que muda (cooldowns, timers, alvo atual) vive num objeto de runtime por personagem, não numa cópia do asset.

depois// O SO nunca é clonado nem mutado. É só a receita.
public class Ability : ScriptableObject { /* dados imutáveis */ }

// O estado de runtime vive à parte, por personagem:
public class AbilityRuntime
{
    public Ability data;      // referência partilhada (não clonada)
    public float cooldownEndTime;
    // ... o que muda durante o jogo
}

O mesmo princípio resolve o StateMachine: os AIState tornam-se sem-estado (recebem tudo por parâmetro ou lêem de um pequeno "blackboard" no controlador), e deixas de precisar de os clonar. Repara que o teu StateCombat guarda lastAttackTime dentro do SO — é exatamente por isso que sentiste a necessidade de Instantiate. Move esse lastAttackTime para o controlador e o clone deixa de ser preciso.

Regra geral

Nunca guardes estado mutável de runtime dentro de um ScriptableObject. Se um SO tem um campo que muda durante o jogo, ou o clonas (caro) ou tens bugs de estado partilhado. A saída é separar "receita" (SO) de "estado" (objeto de runtime).

5. Bug de linha de visão no Perception fazer já

Assets/Scripts/Perception.cs 
antesif (Physics.Raycast(pos, dir.normalized, dist, ~0, QueryTriggerInteraction.Ignore))
{
    return true;   // "vejo o jogador"
}
return false;
A lógica está invertida

Este raycast usa ~0 (bate em tudo) e devolve true se acertar em qualquer coisa dentro da distância. Ou seja: se houver uma parede a tapar o jogador, o raio bate na parede e a IA conclui que "vê o jogador". É o contrário do que queres. Uma linha de visão correta devia: disparar o raio e confirmar que o primeiro que acerta é o jogador (ou que nada opaco o bloqueia).

depois// Bate só em camadas que bloqueiam visão; se o raio chega "limpo" ao jogador, vê-o.
if (Physics.Raycast(pos, dir.normalized, out RaycastHit hit, dist, obstacleMask,
                    QueryTriggerInteraction.Ignore))
{
    // Bateu num obstáculo ANTES do jogador → visão bloqueada.
    return hit.collider.GetComponentInParent<CharacterManager>() == player;
}
return true;   // nada bloqueou → vê o jogador

Nota: define uma LayerMask obstacleMask no Inspector com o que bloqueia visão (paredes, props), em vez de ~0. Mais barato e correto.

6. Strings mágicas de animação → hashes a seguir

Espalhados pelo código: animator.Play("Death"), animator.SetFloat("Speed", ...). Strings mágicas não têm verificação em compilação (um typo = bug silencioso) e são re-hashadas a cada chamada.

antescharacterManager.animator.Play("Death");
animator.SetFloat("Speed", agent.desiredVelocity.magnitude, 0.1f, Time.deltaTime);
depoispublic static class AnimHashes
{
    public static readonly int Death = Animator.StringToHash("Death");
    public static readonly int Speed = Animator.StringToHash("Speed");
}
// uso:
animator.Play(AnimHashes.Death);
animator.SetFloat(AnimHashes.Speed, speed, 0.1f, Time.deltaTime);

Já tens um enum AnimationName usado no MeleeAttack — bom instinto. Estende essa ideia a todas as animações e parâmetros, e usa hashes (int) em vez de strings nas chamadas quentes.

7. Dependência escondida: FindAnyObjectByType a seguir

Assets/Scripts/CharacterManager.cs 
antespublic PlayerInput GetPlayerInput()
{
    if (_playerInput == null)
        _playerInput = FindAnyObjectByType<PlayerInput>(FindObjectsInactive.Include);
    return _playerInput;
}

Cada CharacterManager (jogador e inimigos) procura o PlayerInput na cena. Funciona, mas é uma dependência descoberta em runtime, lenta e invisível. Como o IsPlayer() e o GetPlayer() dependem disto, muita coisa passa por aqui.

O "depois" pragmático (sem framework de DI)

Um Service Locator minúsculo e explícito resolve, sem a complexidade de injeção de dependências a sério:

depoispublic static class GameContext
{
    public static CharacterManager Player { get; set; }   // definido pelo PlayerInput no arranque
}
// CharacterManager fica:
public bool IsPlayer() => GameContext.Player == this;
public CharacterManager GetPlayer() => GameContext.Player;

Uma referência, definida uma vez quando o jogador aparece (ou quando trocas de personagem no teu PlayerInput.SetCharacter). Sem Find por objeto, sem custo por chamada. Nota: isto continua a ser estado global, mas uma referência simples e explícita não é o mesmo que um event bus — é aceitável e comum. Se preferires, podes injetar a referência nos inimigos quando a factory os cria (secção 10 da nota senior).

8. Duplicação: movimento, distâncias, cooldown a seguir

8.1 Movimento relativo à câmara — calculado duas vezes

Em PlayerInput, o bloco "camForward/camRight, zerar y, normalizar, combinar" aparece igual em HandleMovement() e em GetDodgeDirection(). Extrai um helper GetCameraRelativeInput() e chama-o nos dois sítios. Uma fonte de verdade.

8.2 Distâncias com raiz quadrada desnecessária

StateChase e StateCombat usam Vector3.Distance(...) (que faz sqrt) só para comparar com um limite. O teu LockOn já faz certo com sqrMagnitude. Uniformiza: compara quadrados quando só queres saber "mais perto que X".

8.3 Cooldown de ataque em dois modelos diferentes

A IA tem cooldown em StateCombat (Time.time >= lastAttackTime + attackCooldown). O jogador, no plano de auto-ataque, teria outro timer. Unifica: o ritmo de ataque devia vir da arma (como discutimos), e o mesmo componente de "quando posso atacar?" servir jogador e IA. Coerência de mundo (armas partilhadas) e menos código.

Princípio DRY com juízo

DRY (Don't Repeat Yourself) não é "nunca duplicar" — é "não duplicar conhecimento". Os três casos acima são o mesmo conhecimento em dois sítios: se mudares a regra num, tens de lembrar-te do outro. É esse o perigo, não a repetição visual.

9. O isBusy como máquina de estados implícita quando doer

O booleano isBusy é ligado/desligado em muitos sítios: CombatManager.Attack, Dodge, Health.Die, CharacterManager.OnIdle. É uma máquina de estados implícita espalhada — o género de coisa que gera bugs "o personagem ficou preso sem se mexer" difíceis de reproduzir.

Não faças já — mas fica atento

Enquanto for só "livre vs ocupado", o booleano chega. Quando começares a ter estados de ação do jogador (Atacar / Esquivar / Atordoado / Morto) com regras de transição, promove isto a uma máquina de estados explícita para o personagem (como já tens para a IA). O sinal de que chegou a hora: quando te vires a escrever if (!isBusy && !isDodging && !isStunned && ...).

10. A foundation alvo

Organização de pastas sugerida

À medida que isto cresce, agrupa por domínio, não por tipo de ficheiro:

Assets/Scripts/ ├── Core/ GameContext, AnimHashes, helpers ├── Combat/ DamageType (SO), DamageResistance, DamageRequest, DamageResult, │ IDamageable, CombatManager, Hitbox, AutoAttackController ├── Characters/ CharacterManager, Health, DeathReactor, LootDropReactor... ├── Abilities/ Ability (SO), AbilityRuntime, MeleeAttack, DodgeAbility ├── AI/ StateMachine, AIState, State*, Blackboard ├── Items/ WeaponItem, BuffItem, MineralItem, CharacterClass └── Input/ PlayerInput

Ordem de execução recomendada

  1. DamageType (ScriptableObject) + um int damage (secção 1) — corrige o bug de reduções e é a base de tudo o resto.
  2. DamageRequest + IDamageable (secção 2) — desacopla o fluxo de dano.
  3. Health estado puro + 1 DeathReactor (secção 3) — separa responsabilidades.
  4. Tirar estado dos SOs (secção 4) — lastAttackTime fora do StateCombat, elimina os Instantiate.
  5. Corrigir o Perception (secção 5) — é um bug funcional, rápido de resolver.
  6. Depois: hashes de animação, GameContext, deduplicação (secções 6–8).
Aviso senior (o mais importante)

Não faças isto tudo de uma vez num "big bang refactor" — é a forma mais rápida de partir um projeto que funciona. Faz um ponto de cada vez, testa em Play mode, e só avanças quando estiver estável. Cada secção acima é independente por design, precisamente para poderes parar a meio sem deixar o jogo partido. Um refactor que não podes interromper é um refactor mal planeado.
