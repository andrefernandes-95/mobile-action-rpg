Arquitetura de Combate — nível senior

Como um engenheiro experiente (à la FromSoftware) pegaria nos sistemas de combate, classes e itens de As Profundezas e os transformaria numa base épica, extensível e testável. Só ideias e arquitetura — contratos, padrões e diagramas, não implementação para colar.

O que muda quando pensas como senior

Um júnior pergunta "como faço isto funcionar?". Um senior pergunta "como faço isto sobreviver a 100 horas de conteúdo, 200 inimigos no ecrã e 6 meses de mudanças de design sem virar spaghetti?". A resposta quase nunca é "mais código" — é fronteiras mais claras. Este documento é sobre onde traçar essas fronteiras.

1. Os 5 princípios que guiam tudo

Antes dos padrões, a filosofia. Todos os padrões abaixo são só consequências destes princípios:

  1. Composição > herança. Um inimigo não "é um" tipo rígido; é um saco de componentes e dados. Queres um Orc que lança fogo? Compões, não crias uma subclasse FireOrc.
  2. Dados > código. O máximo de comportamento possível vive em ScriptableObjects e assets. Adicionar conteúdo = criar dados, não recompilar.
  3. Desacoplamento por contratos. Os sistemas falam por interfaces e eventos, não por referências concretas. O combate não sabe que existe UI; a UI escuta o combate.
  4. Determinismo onde importa. A lógica de combate deve ser previsível e testável fora de cena — se depende de Update() e de FindObjectOfType, não consegues testar nem depurar.
  5. O custo certo. Cada abstração paga-se com complexidade. Só a adicionas quando a dor de não a ter já se sente. (Ver secção 11 — isto é tão importante como as outras quatro.)

2. A tua pergunta 1: a Health deve "receber chamadas para deduzir vida"?

Resposta curta: sim, mas a tua Health atual faz demasiado. Vamos olhar para o que tens hoje:

Assets/Scripts/Health.cs (atual) 
void Die()
{
    characterManager.animator.Play("Death");        // (A) sabe de animação
    characterManager.agent.ResetPath();             // (B) sabe de navegação
    characterManager.agent.enabled = false;
    characterManager.isBusy = true;                 // (C) sabe de estado de combate
    var player = characterManager.GetPlayer();
    player.lockOn.UnregisterChasingEnemy(...);      // (D) sabe do lock-on do player
}
O "code smell"

A Health devia responder a uma pergunta: "quanta vida tenho?". Mas o teu Die() conhece animação, navegação, estado de combate e o lock-on do jogador. É a God Method em miniatura: sempre que adicionares uma reação à morte (dropar loot, dar XP aos heróis, tocar som, screen shake), este método cresce e fica impossível de testar. Analogia front-end: é um handler de onClick que, além de tratar o clique, também faz fetch, atualiza o router, escreve no localStorage e manda analytics. Toda a gente sabe que isso é mau — aqui é o mesmo.

2.1 A separação senior: Health = estado puro + eventos

A Health passa a ser burra e honesta: guarda números e anuncia o que aconteceu. Quem se importa que reaja.

conceito — Health como fonte de estado + sinais 
public sealed class Health : MonoBehaviour
{
    public int Current { get; private set; }
    public int Max { get; private set; }
    public bool IsDead => Current <= 0;

    // Sinais: quem quiser reagir, subscreve. A Health não conhece ninguém.
    public event System.Action<DamageResult> OnDamaged;
    public event System.Action<int> OnHealed;
    public event System.Action OnDied;

    public void Apply(in DamageResult result)
    {
        if (IsDead) return;
        Current = Mathf.Max(0, Current - result.FinalDamage);
        OnDamaged?.Invoke(result);
        if (Current == 0) OnDied?.Invoke();
    }
}

Agora a morte "épica" monta-se com reatores independentes, cada um com uma responsabilidade. O animator subscreve OnDied. O loot subscreve OnDied. O lock-on do jogador subscreve OnDied. Nenhum sabe dos outros:

Health.OnDied ├── DeathAnimationReactor → toca "Death", desliga o agent ├── LootDropReactor → instancia ouro/minério/arma ├── LockOnReactor → remove-se da lista de ameaças do player ├── AudioReactor → som de morte └── ProgressionReactor → (ironia do GDD) dá XP ao herói que te matou
O ganho concreto

Adicionar "dropar loot ao morrer" deixa de tocar na Health. Crias um LootDropReactor, subscreves OnDied, e pronto. Isto é o Open/Closed Principle: aberto a extensão, fechado a modificação. E cada reator é testável isoladamente.

2.2 Mas quem calcula o dano, então?

Repara que a Health.Apply() recebe um DamageResult já calculado. A Health não decide reduções, críticos ou i-frames — isso é trabalho do pipeline (secção 4). A Health é só o sink final. Esta é a fronteira que muda tudo: separar "quanto dano" de "aplicar o dano".

3. Stat system com modificadores Decorator / Strategy

O teu design tem: ouro que sobe atributos, upgrades de arma, buffs que "alteram mecânicas", classes com stats base. Se cada um destes escrever directamente em campos (health += 10), vais ter bugs impossíveis: como remover exatamente o bónus de um anel quando o tiras, se três coisas mexeram no mesmo número?

A solução clássica de RPG: um stat é uma base + uma pilha de modificadores. O valor final é calculado, nunca escrito.

conceito — stats calculados a partir de modificadores 
public enum StatType { MaxHealth, AttackPower, AttackSpeed, MoveSpeed, Armor }

public enum ModifierOp { Flat, PercentAdd, PercentMult }

public readonly struct StatModifier
{
    public readonly StatType Type;
    public readonly ModifierOp Op;
    public readonly float Value;
    public readonly object Source;   // quem o adicionou (anel? arma? classe?)
}

// Valor final = (base + soma dos Flat) * (1 + soma dos PercentAdd) * produto dos PercentMult
Porque três operações e não uma?

A ordem de aplicação define o "feeling" do balanceamento. Flat primeiro (armadura +5), depois PercentAdd (buffs somam-se: +10% e +10% = +20%), e PercentMult por último e multiplicativos entre si (um buff lendário "×1.5" que não se dilui somando-se aos outros). Isto é exactamente como Path of Exile e afins evitam que 10 bónus de +10% cheguem trivialmente a +100%.

Remover um anel é agora trivial e correcto: filtras os modificadores cujo Source é esse anel e recalculas. Sem drift, sem bugs de "o número não voltou ao valor original".

4. O Damage Pipeline Chain of Responsibility

Aqui é onde o combate fica épico e extensível. Em vez de um if gigante que trata i-frames, reduções, críticos, lifesteal, stagger e afins (o teu DamageReceiver.ApplyDamage já começa a acumular isto), modelas o dano como uma requisição que atravessa uma cadeia de estágios.

conceito — o contrato do dano 
public readonly struct DamageRequest
{
    public readonly CharacterManager Source;
    public readonly DamageType Type;       // ScriptableObject (não enum) — ver Refactor-Foundation
    public readonly int Amount;
    public readonly Vector3 HitPoint;
    public readonly float Poise;           // para stagger / hyperarmor
}

public struct DamageContext          // muta ao longo da cadeia
{
    public DamageRequest Request;
    public IDamageable Target;
    public float RunningDamage;
    public bool IsCritical;
    public bool Cancelled;             // um estágio pode abortar (ex.: dodge)
}

public interface IDamageStage { void Process(ref DamageContext ctx); }

E a cadeia é só uma lista ordenada de estágios. Cada um faz uma coisa:

DamageRequest ▼ [1] InvulnerabilityStage → se target.IsDodging: ctx.Cancelled = true (para tudo) [2] ResistanceStage → aplica reduções por tipo (o teu código atual) [3] CriticalStage → rola crítico, marca ctx.IsCritical, multiplica [4] BuffHookStage → lifesteal, dano bónus vs "heróis", etc. [5] PoiseStage → acumula stagger; decide se há hitstun [6] ApplyStage → Health.Apply(new DamageResult(ctx)) ▼ DamageResult (para os eventos/UI)
Porque isto é "FromSoftware-tier"

Queres adicionar "o Orc causa +50% de dano a heróis com armadura pesada"? Escreves um estágio novo e insere-lo na lista — sem tocar em nada existente. Queres uma arma que ignora armadura? Um estágio que zera as reduções. Queres um boss com hyperarmor (não sofre stagger)? Configura o PoiseStage. Cada regra de combate é uma peça Lego isolada, ordenável e testável. É esta a diferença entre um combate que cresce e um que apodrece.

Onde vive a cadeia? Pode ser por personagem (montada a partir da classe + arma + buffs equipados) ou global com estágios que se auto-configuram do contexto. Começaria por-personagem: é mais fácil de raciocinar e cada inimigo pode ter a sua receita.

5. A tua pergunta 2: Event Bus — sim, mas com regras Observer / Mediator

Perguntaste se fazia sentido um event bus. Faz — mas o erro clássico é mandar tudo pelo bus e transformar o jogo numa teia invisível onde ninguém sabe quem dispara o quê. A regra senior é uma linha divisória clara:

Usa chamada direta / evento local quando…Usa o Event Bus global quando…
A lógica é crítica e ordenada (o pipeline de dano, aplicar vida). Precisas de determinismo e de saber a ordem exata. É uma notificação transversal "aconteceu X, quem quiser que reaja" (UI, áudio, conquistas, tutorial, estatísticas).
O emissor precisa de uma resposta ou de garantir que algo aconteceu. O emissor não quer saber quem escuta (nem se alguém escuta).
Ex.: Hitbox → pipeline → Health.Apply(). Ex.: EnemyDied, BossPhaseChanged, PlayerLeveledUp, WeaponUpgraded.

5.1 Um bus tipado (sem strings mágicas)

Nada de Raise("enemy_died") com strings — é frágil e sem verificação em compilação. Um bus genérico por tipo de evento dá-te segurança de tipos e auto-complete:

conceito — event bus tipado 
public interface IEvent { }

public static class EventBus<T> where T : IEvent
{
    static event System.Action<T> handlers;
    public static void Subscribe(System.Action<T> h)   => handlers += h;
    public static void Unsubscribe(System.Action<T> h) => handlers -= h;
    public static void Raise(in T e)                    => handlers?.Invoke(e);
}

// Eventos são structs pequenos e imutáveis:
public readonly struct EnemyDied : IEvent
{
    public readonly CharacterManager Enemy;
    public readonly CharacterManager Killer;
    public readonly Vector3 Position;
}

// Emitir (do LootDropReactor, por ex.):  EventBus<EnemyDied>.Raise(new EnemyDied(...));
// Escutar (na UI):                        EventBus<EnemyDied>.Subscribe(OnEnemyDied);
As 3 armadilhas do event bus (todas te vão morder)
  • Leaks de subscrição. Se subscreves em OnEnable e não te dessubscreves em OnDisable, o handler fica vivo depois do objeto morrer → NullReferenceException fantasma ou objetos que não são libertados. Regra: toda subscrição tem a sua dessubscrição simétrica.
  • Ordem indefinida. Se dois sistemas precisam de reagir por uma ordem específica, o bus é a ferramenta errada — usa uma cadeia explícita. O bus é para reações independentes.
  • Debugging invisível. Um Raise pode acordar 10 sistemas que não vês no stack trace. Mitiga com um modo de log que imprime cada evento e os seus subscritores. Um bus sem observabilidade é uma dor de cabeça garantida.

Bónus: para o combate de um encontro específico (ex.: uma boss fight), um bus com escopo (instância, não estático) que morre no fim do encontro evita muitos destes problemas — o estado não sobrevive ao encontro.

6. Command + input buffering Command

O teu combate é auto-ataque, mas o feeling soulslike vem do dodge com timing. Em mobile, com lag de toque, "engolir" inputs mata a sensação. Os jogos da From resolvem isto com input buffering: um input é um objeto (comando) que entra numa fila com timestamp e é consumido dentro de uma janela.

conceito — comandos em buffer 
public interface ICombatCommand
{
    void Execute(CharacterManager cm);
}

public readonly struct DodgeCommand : ICombatCommand { /* ... */ }

// O buffer guarda (comando, tempo). Quando o personagem fica livre,
// consome o comando mais recente ainda dentro da janela (ex.: 0.2s).
// Resultado: carregas em dodge um bocadinho antes de poderes, e ele
// dispara no instante exato em que a animação anterior liberta o controlo.
Porque comando e não um bool?

Modelar o input como objeto dá-te de graça: fila (buffering), histórico (para combos ou replay), rede (enviar comandos em vez de estado), e teste (injetar comandos sem tocar no ecrã). Um bool dodgePressed não te dá nada disto.

7. IA: de FSM para HFSM / Behaviour Tree + Blackboard State / Behaviour Tree / Blackboard

Já tens uma máquina de estados (StatePatrol, StateChase, StateCombat). Para inimigos comuns é perfeita — não mexas. Para bosses (os teus heróis-clichê), uma FSM plana explode em transições ("de qualquer estado, se vida<30%, ir para Enrage" × N estados). Duas evoluções:

  • HFSM (hierárquica): estados dentro de estados. "Combate" contém "Aproximar / Atacar / Recuar"; uma transição no pai (ex.: Enrage) aplica-se a todos os filhos. Mata a explosão de transições.
  • Behaviour Tree: para padrões de boss ricos (sequências, seletores, decoradores de probabilidade). É o standard da indústria para IA de combate porque compõe comportamento a partir de nós reutilizáveis — muito parecido com compor uma árvore de componentes.

Ambos precisam de um Blackboard: um quadro de dados partilhado (alvo atual, última posição vista, cooldowns) que os estados/nós leem e escrevem, em vez de se conhecerem uns aos outros. É o "estado partilhado" da IA, desacoplado.

Ligação ao GDD

Cada herói tem um padrão distinto (Willkard chama reforços, Cassandra usa ilusões, Hroth quebra guarda). Uma Behaviour Tree por boss, montada de nós partilhados (MoveToTarget, TelegraphAttack, SummonAdds), deixa-te criar um boss novo compondo nós — dados, não código novo.

8. Bullet-heaven a sério: pooling e data-oriented Object Pool / DOD / Flyweight

Vampire Survivors = centenas de inimigos e projéteis. Aqui a arquitectura "bonita" orientada a objetos bate na parede da performance, sobretudo em mobile. Três frentes:

8.1 Object Pooling — nunca Instantiate/Destroy em loop

Criar e destruir GameObjects por frame gera lixo para o Garbage Collector → stutter. Pré-alocas um pool e reutilizas. (Isto liga directamente à conversa que já tivemos sobre NonAlloc e evitar alocações por frame — é o mesmo princípio, ao nível dos objetos.)

8.2 Data-Oriented Design para as hordes

Para os inimigos comuns "burros" (andar em direção ao jogador, dano por colisão), um MonoBehaviour completo por inimigo é caro. A abordagem senior: um sistema que itera arrays de dados (posições, velocidades, vida) em vez de N objetos com Update() cada. É a mentalidade do Unity DOTS/ECS. Não precisas de adotar DOTS já — mas separa "inimigo-horda" (dados simples, atualizados em massa) de "inimigo-elite/boss" (objeto rico com IA própria).

Inimigo-horda → dados em arrays, 1 sistema atualiza todos (barato, aos milhares) Inimigo-elite → MonoBehaviour + FSM (dezenas) Boss → Behaviour Tree + blackboard (1)

8.3 Partição espacial

"Que inimigos estão perto do jogador?" com 300 inimigos não se resolve com um loop O(n²). Uma grelha espacial (spatial hash) responde em O(1) por célula. É o que permite auto-ataque de área e lock-on baratos com muitos alvos.

9. Composição, Factories e injeção de dependências Factory / Builder / DI

9.1 Factory para spawns

"Nascer um inimigo" não devia ser Instantiate espalhado por 5 sítios. Uma EnemyFactory centraliza: tira do pool, aplica a CharacterClass, monta o pipeline de dano, liga a IA. Um sítio para mudar, um sítio para testar.

9.2 O problema do FindObjectOfType

O teu CharacterManager.GetPlayerInput() faz FindAnyObjectByType. Funciona, mas é uma dependência escondida e lenta. A alternativa senior é injetar dependências (passá-las na construção/inicialização) ou usar um Service Locator explícito (Services.Get<PlayerContext>()). A diferença: as dependências ficam visíveis e substituíveis (essencial para testes e para o modo em que o "jogador" é um inimigo com a mesma lógica).

Regra prática

Se um sistema faz Find... ou GetComponent repetidamente em runtime para descobrir com quem falar, isso é uma dependência que devia ser declarada, não descoberta. Descobre uma vez, guarda a referência, ou injeta-a.

11. Aviso: quando NÃO fazer nada disto o princípio mais senior de todos

A marca real de um senior

Não é conhecer estes padrões — é saber quando não os usar. Cada abstração acima é uma dívida de complexidade. Aplicada cedo demais, atrasa-te e esconde bugs atrás de camadas. A regra: implementa a versão simples primeiro; refatora para o padrão quando a dor de não o ter for real.

Faz já (o retorno é imediato): separar a Health em estado + eventos; um Damage Pipeline mínimo (mesmo com 2-3 estágios); object pooling para inimigos e projéteis. Estes pagam-se no primeiro mês.
Faz quando doer: event bus global (quando tiveres 4+ sistemas a querer reagir ao mesmo acontecimento); Behaviour Trees (quando o primeiro boss ficar impossível de gerir com FSM); stat system completo (quando tiveres muitas fontes de bónus a colidir).
Provavelmente não precisas (ainda): DOTS/ECS a sério (só se o profiler gritar com milhares de inimigos); injeção de dependências com framework (um Service Locator simples chega para muito longe); rede/replay (a não ser que seja um objetivo do produto).
🔢 Para investigar (matemática)

O stat system multiplicativo levanta uma pergunta gira. Se um buff dá ×1.1 e empilhas n deles multiplicativamente, o total é 1.1ⁿ — crescimento exponencial. Com aditivos, n buffs de +10% dão 1 + 0.1n — crescimento linear.

  • Para n = 10 buffs, quanto dá cada modelo? Porque é que o exponencial é perigoso para o balanceamento de late-game?
  • A partição espacial (secção 8.3) troca uma pesquisa O(n²) por algo perto de O(n). Para n = 300 inimigos, quantas comparações poupas por frame? (Compara n² com n.)
Desafio final

Escolhe um sistema deste documento e desenha, em papel, os seus contratos (interfaces + eventos) antes de escrever código. Se conseguires explicar as fronteiras a outra pessoa em 2 minutos, o desenho está bom. Se precisares de 10, ainda está acoplado demais. Qual escolhes — o Damage Pipeline ou a Health event-driven?
