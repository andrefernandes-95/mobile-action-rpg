As Profundezas — Combate, Classes e Itens

Documento de inspiração: código ilustrativo em C# para o combate estilo Vampire Survivors + Soulslike, classes de personagem e itens (armas, buffs, minérios). Pensado para leres, perceberes e implementares tu próprio.

Lê primeiro

Este ficheiro não é para colar directamente no projecto. É um mapa de ideias com código de exemplo. Cada bloco existe para explicar uma decisão de arquitectura — o objectivo é dar-te intuição para depois escreveres a tua própria versão, coerente com o que já tens.

As convenções seguem o teu projecto real: namespace AF, habilidades como ScriptableObject (tal como Ability, MeleeAttack, DodgeAbility), e o CharacterManager como hub central de referências.

1. O que já tens (mapa mental)

Antes de adicionar seja o que for, vale a pena ver o fluxo de combate que já existe no projecto. Pensa nisto como a "árvore de componentes" de uma app front-end: há um componente-raiz (CharacterManager) que segura referências para os filhos especializados.

CharacterManager (o "componente-raiz" — segura referências) ├── Health vida + morte ├── Hitbox volume que causa dano quando "aberto" ├── CharacterDamageReceiver recebe dano (com reduções por tipo) ├── Dodge esquiva com i-frames ├── Perception deteta inimigos ├── LockOn escolhe o alvo mais próximo └── CombatManager orquestra as habilidades Fluxo de um ataque hoje (manual): CombatManager.Attack() → SetCurrentAbility(attackAbility) // Instantiate do ScriptableObject → MeleeAttack.OnStart() // toca animação, subscreve eventos → (animação abre a Hitbox) → Hitbox.OpenHitbox() // dispara onHitboxOpen → OnTriggerStay deteta alvo → DamageReceiver.ApplyDamage() // calcula dano por tipo → Health.TakeDamage()
A observação-chave

O teu ataque já é dirigido por dados: o MeleeAttack é um ScriptableObject com os valores de dano. Isto é exactamente o alicerce certo para Vampire Survivors — só falta uma peça: quem carrega no botão de atacar. Hoje é manual (CombatManager.Attack()); vamos torná-lo automático e ritmado.

2. Sistema de combate — auto-ataque com timing

O requisito: "a arma ataca automaticamente de X em X tempo (configurável por arma e skills)" e "quando um inimigo está presente e junto a nós, ficamos em lock-on virados para ele". Boas notícias: o lock-on já faz metade disto. Falta o relógio que dispara o ataque.

2.1 A ideia: um temporizador que "carrega no botão" por ti

Analogia front-end: é como um setInterval que, a cada tick, verifica "há alvo válido?" e, se sim, invoca a acção de ataque. A diferença é que em Unity não usamos setInterval — descontamos Time.deltaTime a um contador dentro do Update(), para respeitar pausas e a escala de tempo do jogo.

Assets/Scripts/Combat/AutoAttackController.cs novo 
namespace AF
{
    using UnityEngine;

    // Vive no mesmo GameObject do CharacterManager (jogador E inimigos podem usá-lo).
    public class AutoAttackController : MonoBehaviour
    {
        [SerializeField] CharacterManager characterManager;

        [Header("Arma equipada (dados)")]
        [SerializeField] WeaponItem equippedWeapon;   // ver secção 4

        [Header("Alcance de deteção do alvo")]
        [SerializeField] float engageRange = 2.5f;

        float attackTimer;

        void OnEnable()
        {
            // Começa "pronto a atacar" para o primeiro golpe não ter delay artificial.
            attackTimer = 0f;
        }

        void Update()
        {
            if (equippedWeapon == null) return;
            if (characterManager.health.IsDead()) return;
            if (characterManager.isBusy) return;      // já está a meio de uma ação

            // 1) O relógio corre sempre, mesmo sem alvo — assim o primeiro golpe é imediato.
            if (attackTimer > 0f)
                attackTimer -= Time.deltaTime;

            // 2) Sem alvo válido em alcance? Não gastamos o ataque.
            if (!HasTargetInRange()) return;

            // 3) Ainda em cooldown? Espera.
            if (attackTimer > 0f) return;

            // 4) Ataca e rearma o relógio com o intervalo DA ARMA.
            characterManager.combatManager.Attack();
            attackTimer = equippedWeapon.GetAttackInterval();
        }

        bool HasTargetInRange()
        {
            LockOn lockOn = characterManager.lockOn;
            if (lockOn == null || !lockOn.isLockedOn || lockOn.lockOnTarget == null)
                return false;

            float sqrDist =
                (lockOn.lockOnTarget.position - transform.position).sqrMagnitude;

            return sqrDist <= engageRange * engageRange;
        }
    }
}
Porque comparar sqrMagnitude e não Distance?

Calcular uma distância real precisa de uma raiz quadrada (lenta). Comparar o quadrado da distância com o quadrado do alcance dá o mesmo resultado booleano sem a raiz. É o mesmo truque que o teu LockOn já usa (repara no closestSqrDist). Micro-otimização, mas grátis e idiomática.

2.2 O "X em X tempo configurável por arma e skills"

Repara na linha attackTimer = equippedWeapon.GetAttackInterval();. O intervalo não está no controlador — está na arma. É isso que torna cada arma diferente: um machado pesado de Orc pode ter intervalo 1.4s, uma adaga 0.4s. E as "skills" podem reduzir esse intervalo (ver secção 4 sobre buffs).

Ponto de fricção que discutimos

No GDD anotámos que inimigos e jogador partilham armas/habilidades. Este AutoAttackController foi desenhado a pensar nisso: como lê tudo do CharacterManager e da WeaponItem, o mesmo script serve para um esqueleto inimigo ou para o teu Orc. A diferença é só que dados lá pões. Coerência de mundo com custo de código zero — mas lembra-te: cada arma tem agora de ser balanceada nos dois lados.

2.3 Uma pequena melhoria ao CombatManager

Hoje o teu CombatManager tem a attackAbility fixada no Inspector. Se a arma é que manda, faz sentido a arma fornecer a habilidade de ataque. Uma forma minimamente invasiva:

Assets/Scripts/CombatManager.cs alteração sugerida 
// ... dentro de CombatManager, além do que já tens:

public void EquipWeapon(WeaponItem weapon)
{
    // A arma traz a sua própria habilidade de ataque (um MeleeAttack, por ex.).
    attackAbility = weapon.attackAbility;
}

Assim, trocar de arma passa a ser "trocar os dados", e o AutoAttackController nem precisa de saber que a arma mudou — continua a chamar combatManager.Attack(). Baixo acoplamento: cada peça sabe o mínimo sobre as outras.

2.4 E o "timing é essencial"? (a alma soulslike)

Se o ataque é automático, onde entra a perícia do jogador? Nas duas coisas que ficam manuais: mover-se e esquivar. O teu DodgeAbility + os i-frames do CharacterDamageReceiver (repara: IsInvulnerable() devolve characterManager.dodge.isDodging) já são a base. O jogador não escolhe quando ataca; escolhe onde está e quando fica invulnerável. É o mesmo contrato de Vampire Survivors, mas com a leitura de padrões de um Souls.

Auto-ataque → tira a decisão de "quando bater" (ritmo constante) Dodge + i-frames → devolve a decisão de "quando não levar" (timing) Posicionamento → a terceira decisão contínua (alcance, hordas)

3. Classes de personagem

Requisito: "personagem principal deriva de heróis pré-feitos com classes predefinidas; o Orc usa armas pesadas (machados, grandes espadas)". Traduzido: cada classe é um conjunto de dados — vida base, poder base, que tipos de arma pode usar e com que arma começa.

3.1 Um enum para tipos de arma

Assets/Scripts/Items/WeaponType.cs novo 
namespace AF
{
    public enum WeaponType
    {
        Heavy,   // machados, grandes espadas — o Orc
        Sword,
        Dagger,
        Staff,
        Bow
    }
}

3.2 A classe como ScriptableObject

Porquê ScriptableObject outra vez? Porque uma "classe" é dados partilhados e imutáveis — como um módulo de configuração que importas. Todos os Orcs partilham a mesma definição de classe; não faz sentido cada instância carregar a sua cópia. (Front-end: é a diferença entre uma constante exportada de um módulo e o estado local de um componente.)

Assets/Scripts/Classes/CharacterClass.cs novo 
namespace AF
{
    using UnityEngine;

    [CreateAssetMenu(menuName = "AF/Data/Character Class")]
    public class CharacterClass : ScriptableObject
    {
        [Header("Identidade")]
        public string className;
        [TextArea] public string description;
        public Sprite portrait;

        [Header("Atributos base")]
        public int baseHealth = 100;
        public int baseAttackPower = 10;
        public float baseMoveSpeed = 3.5f;

        [Header("Restrições de arma")]
        public WeaponType[] allowedWeaponTypes;
        public WeaponItem startingWeapon;

        public bool CanUse(WeaponItem weapon)
        {
            foreach (WeaponType allowed in allowedWeaponTypes)
                if (allowed == weapon.weaponType)
                    return true;
            return false;
        }
    }
}

3.3 Aplicar a classe ao personagem

Quando o personagem nasce, "aplica" a classe: define a vida no Health e equipa a arma inicial. Um pequeno componente trata disto no Awake().

Assets/Scripts/Classes/CharacterClassProfile.cs novo 
namespace AF
{
    using UnityEngine;

    public class CharacterClassProfile : MonoBehaviour
    {
        [SerializeField] CharacterManager characterManager;
        [SerializeField] CharacterClass characterClass;
        [SerializeField] AutoAttackController autoAttack;

        void Awake()
        {
            // Atributos base vindos da classe.
            characterManager.health.health = characterClass.baseHealth;

            // Equipa a arma inicial (se a classe a permitir — deve permitir).
            WeaponItem starting = characterClass.startingWeapon;
            if (starting != null && characterClass.CanUse(starting))
            {
                characterManager.combatManager.EquipWeapon(starting);
                autoAttack.SetWeapon(starting);   // pequeno setter no controlador
            }
        }
    }
}
Exemplo: a classe Orc

No Unity criarias um asset (botão direito → Create → AF → Data → Character Class) e preenchias:

  • className: "Orc"
  • baseHealth: 140 (é resistente)
  • baseAttackPower: 14
  • allowedWeaponTypes: [ Heavy ] — só armas pesadas
  • startingWeapon: o asset "Machado Grande"

Repara que não escreves uma classe C# por cada herói. "Orc", "Goblin", etc. são todos assets do mesmo tipo CharacterClass. Adicionar uma classe nova é criar um ficheiro de dados, não compilar código. É esta a magia do design orientado a dados que o teu projecto já abraça.

4. Itens — armas, buffs e minérios

4.1 Armas novo

A arma junta três coisas: identidade (nome, ícone, tipo), comportamento (a habilidade de ataque) e ritmo/poder (intervalo e escalonamento por upgrade).

Assets/Scripts/Items/WeaponItem.cs 
namespace AF
{
    using UnityEngine;

    [CreateAssetMenu(menuName = "AF/Data/Item/Weapon")]
    public class WeaponItem : ScriptableObject
    {
        [Header("Identidade")]
        public string displayName;
        [TextArea] public string description;
        public Sprite icon;
        public WeaponType weaponType;

        [Header("Comportamento de ataque")]
        public MeleeAttack attackAbility;   // reutiliza o teu Ability existente!

        [Header("Ritmo")]
        public float baseAttackInterval = 1.2f;      // segundos entre golpes
        public float intervalReductionPerLevel = 0.03f;
        public float minInterval = 0.2f;             // teto de velocidade

        [Header("Progressão (ferreiro)")]
        [Range(0, 10)] public int upgradeLevel = 0;
        public int bonusDamagePerLevel = 2;

        public float GetAttackInterval()
        {
            float interval = baseAttackInterval - upgradeLevel * intervalReductionPerLevel;
            return Mathf.Max(minInterval, interval);
        }
    }
}
Armadilha importante: não mutes o ScriptableObject em runtime

Tentador: subir de nível a arma fazendo weapon.upgradeLevel++. Não faças isso. Um ScriptableObject é um asset partilhado — mudá-lo em runtime altera o ficheiro no editor e afecta todas as instâncias e saves. É o equivalente a mutar um objecto importado de um módulo global em JS: parece que funciona, até rebentar.

Solução: guarda o nível de upgrade por save, numa estrutura à parte (ver secção 6), e usa o asset só como "valores base". O upgradeLevel no exemplo acima é para prototipagem rápida — em produção, vem do save.

4.2 Buffs (os "anéis") — no máximo 2 novo

Requisito: "podes ter até dois buffs de uma só vez (alteram mecânicas de jogo, como os anéis em Dark Souls)". A palavra-chave é mecânicas, não só números. Por isso um buff não é um saco de +10 força: é um objecto com OnEquip / OnUnequip que pode mexer no que quiser.

Assets/Scripts/Items/BuffItem.cs 
namespace AF
{
    using UnityEngine;

    // Abstrato: cada buff concreto decide o que faz ao ser equipado.
    public abstract class BuffItem : ScriptableObject
    {
        [Header("Identidade")]
        public string buffName;
        [TextArea] public string description;
        public Sprite icon;

        public abstract void OnEquip(CharacterManager characterManager);
        public abstract void OnUnequip(CharacterManager characterManager);
    }
}

Exemplo de buff concreto

Assets/Scripts/Items/Buffs/SwiftStrikesBuff.cs 
namespace AF
{
    using UnityEngine;

    [CreateAssetMenu(menuName = "AF/Data/Item/Buff/Swift Strikes")]
    public class SwiftStrikesBuff : BuffItem
    {
        [Range(0f, 0.5f)] public float attackSpeedBonus = 0.2f; // 20% mais rápido

        public override void OnEquip(CharacterManager cm)
        {
            var auto = cm.GetComponent<AutoAttackController>();
            if (auto != null) auto.AddIntervalMultiplier(1f - attackSpeedBonus);
        }

        public override void OnUnequip(CharacterManager cm)
        {
            var auto = cm.GetComponent<AutoAttackController>();
            if (auto != null) auto.RemoveIntervalMultiplier(1f - attackSpeedBonus);
        }
    }
}

Isto implica um pequeno acrescento ao AutoAttackController — um multiplicador de intervalo que os buffs empilham. Deixo-te a assinatura como desafio (ver secção 7), mas a ideia é: o intervalo final = weapon.GetAttackInterval() * produtoDosMultiplicadores.

O "loadout" de 2 buffs

Assets/Scripts/Items/BuffLoadout.cs 
namespace AF
{
    using System.Collections.Generic;
    using UnityEngine;

    public class BuffLoadout : MonoBehaviour
    {
        public const int MaxBuffs = 2;   // a regra do design, num só sítio

        [SerializeField] CharacterManager characterManager;
        readonly List<BuffItem> equipped = new();

        public bool Equip(BuffItem buff)
        {
            if (equipped.Count >= MaxBuffs) return false;   // cheio
            if (equipped.Contains(buff)) return false;       // já equipado

            equipped.Add(buff);
            buff.OnEquip(characterManager);
            return true;
        }

        public void Unequip(BuffItem buff)
        {
            if (equipped.Remove(buff))
                buff.OnUnequip(characterManager);
        }

        public IReadOnlyList<BuffItem> Equipped => equipped;
    }
}
Nota de tom (liga ao GDD)

Como os buffs vêm de heróis derrotados, os nomes são espaço para a sátira: um anel "Currículo do Willkard" que dá bónus quando há público a ver, ou "Sede de Combate do Hroth" que acelera o ataque mas baixa a defesa. A mecânica serve a piada e vice-versa.

4.3 Minérios e o ferreiro novo

Requisito: "podes encontrar minérios para melhorar a tua arma permanentemente no ferreiro". O minério é uma moeda de crafting; o ferreiro é onde a gastas para subir o upgradeLevel de uma arma — no save, não no asset (lembra-te da armadilha acima).

Assets/Scripts/Items/MineralItem.cs 
namespace AF
{
    using UnityEngine;

    [CreateAssetMenu(menuName = "AF/Data/Item/Mineral")]
    public class MineralItem : ScriptableObject
    {
        public string displayName;
        public Sprite icon;
        [Tooltip("Raridade influencia quanto contribui para um upgrade.")]
        public int upgradeValue = 1;
    }
}
Assets/Scripts/Systems/Blacksmith.cs 
namespace AF
{
    using UnityEngine;

    public class Blacksmith : MonoBehaviour
    {
        [SerializeField] PlayerProfile profile;   // o save (ver secção 6)

        // Devolve true se conseguiu melhorar.
        public bool UpgradeWeapon(WeaponItem weapon, int mineralCost)
        {
            int currentLevel = profile.GetWeaponLevel(weapon);
            if (currentLevel >= 10) return false;             // já no máximo

            if (profile.mineralCount < mineralCost) return false; // sem recursos

            profile.mineralCount -= mineralCost;
            profile.SetWeaponLevel(weapon, currentLevel + 1);
            return true;
        }
    }
}

5. Como tudo se liga

Vale a pena ver o grafo de dependências antes de escreveres uma linha. Setas = "conhece / usa".

CharacterClass (dados) │ aplica atributos + arma inicial ▼ AutoAttackController ──▶ CombatManager ──▶ Ability/MeleeAttack ──▶ Hitbox ──▶ DamageReceiver ──▶ Health │ lê ritmo ▲ ▼ │ EquipWeapon() WeaponItem (dados) ──────────┘ ▲ │ nível de upgrade vem do save PlayerProfile (save) ◀── Blacksmith gasta minérios ▲ │ BuffLoadout (máx 2) ──▶ BuffItem.OnEquip() ──▶ altera AutoAttackController / stats
Regra de ouro da arquitectura

Os dados (ScriptableObjects: classe, arma, buff, minério) não conhecem ninguém — são inertes. Os comportamentos (MonoBehaviours no personagem) é que leem esses dados e agem. Isto mantém o combate testável e os inimigos a partilharem exactamente o mesmo código do jogador.

6. Persistência e New Game Plus

Requisitos: ouro melhora atributos; armas ficam guardadas e levam-se em NG+; minérios melhoram armas. Tudo isto é estado do jogador, separado dos assets. Uma estrutura central (o "save") resolve.

Assets/Scripts/Systems/PlayerProfile.cs esboço 
namespace AF
{
    using System.Collections.Generic;
    using UnityEngine;

    // Podia ser um ScriptableObject de runtime OU uma classe serializada para JSON.
    // O que importa: é UMA fonte de verdade para o progresso permanente.
    public class PlayerProfile : MonoBehaviour
    {
        [Header("Moedas")]
        public int gold;
        public int mineralCount;

        [Header("Meta-progressão de atributos")]
        public int bonusMaxHealth;
        public int bonusAttackPower;

        [Header("Coleção de armas (persistem em NG+)")]
        public List<WeaponItem> unlockedWeapons = new();

        // Nível de upgrade POR arma — a chave é o asset, o valor é o nível no save.
        readonly Dictionary<WeaponItem, int> weaponLevels = new();

        public int GetWeaponLevel(WeaponItem w)
            => weaponLevels.TryGetValue(w, out int lvl) ? lvl : 0;

        public void SetWeaponLevel(WeaponItem w, int lvl)
            => weaponLevels[w] = lvl;

        public void UnlockWeapon(WeaponItem w)
        {
            if (!unlockedWeapons.Contains(w))
                unlockedWeapons.Add(w);
        }
    }
}
Detalhe de serialização

Um Dictionary não é serializável pelo Unity "out of the box", e guardar referências directas a ScriptableObject num ficheiro de save é frágil (o que acontece se renomeares o asset?). Na prática usarias um id string estável por arma (ex.: "weapon_orc_greataxe") como chave. Fica registado como decisão para quando fores implementar a sério — não deixes o save depender de referências de assets.

NG+ em uma linha

Como as armas e a meta-progressão vivem no PlayerProfile e não na cena, um New Game Plus é literalmente: recomeçar as cenas do jogo mantendo o mesmo PlayerProfile. No GDD demos-lhe até uma justificação narrativa (outro grupo de heróis-clichê atraído pela fama das Profundezas).

7. Próximos passos e exercícios

Sugestão de ordem de implementação — do que dá valor mais depressa para o mais acessório:

  1. AutoAttackController — mete o combate a "jogar-se sozinho" com uma arma fixa. É o coração Vampire Survivors.
  2. WeaponItem + EquipWeapon — troca de armas a mudar o ritmo/dano.
  3. CharacterClass — o Orc com vida alta e só armas pesadas.
  4. BuffLoadout + 1 buff simples — sente o "altera mecânicas".
  5. PlayerProfile + Blacksmith — fecha o ciclo de progressão permanente.
Desafios (tenta antes de veres a solução noutro lado)
  • Implementa AddIntervalMultiplier / RemoveIntervalMultiplier no AutoAttackController. Dica: guarda uma lista de multiplicadores e multiplica-os todos ao calcular o intervalo final. Porque é que remover por valor pode ser traiçoeiro se dois buffs derem o mesmo multiplicador?
  • Como impedirias o jogador de equipar uma arma que a classe não permite, na UI, em vez de só no código? (Pensa em CharacterClass.CanUse a pintar o botão a cinzento.)
  • O auto-ataque dispara em lock-on. E se quiseres uma arma de área que ataca mesmo sem alvo fixo (mais Vampire Survivors puro)? Que campo acrescentarias à WeaponItem?
🔢 Para investigar (matemática)

O intervalo de ataque com upgrades é uma função linear: intervalo(n) = base − n · redução, travada por um mínimo. Duas perguntas para pensar:

  • A partir de que nível n a arma atinge o minInterval? (Resolve base − n·redução = minInterval em ordem a n.)
  • Se em vez de subtracção linear usasses multiplicação (intervalo = base · 0.97ⁿ), como mudaria a sensação de progressão? Qual das duas nunca chega a zero — e porque é que isso pode ser desejável para o balanceamento?
